{{ icon('shield', size=18) }} Reverse Proxy SSO（Kerberos / SPNEGO）

已加入 AD 網域、且在 Windows 已登入的使用者，經前端 Nginx（SPNEGO/Kerberos）驗證後， 由 Nginx 傳入使用者帳號標頭，自動登入本系統。非網域電腦或未帶標頭者，仍回到 /login 使用 Local / LDAP / AD / OIDC / SAML 登入。使用者查詢與同步沿用下方「認證設定」的 LDAP / AD 連線。完整部署（AD service account / setspn / ktpass / 瀏覽器設定）詳見文件 reverse_proxy_sso.md。

{{ icon('shield', size=15) }} 反向代理主機（Nginx）必做設定 — 少任一項會不安全或無法登入
  1. Nginx 端做 SPNEGO / Kerberos 驗證，並「一律覆寫」帳號標頭：在 location 內加 proxy_set_header X-Remote-User $remote_user; —— 先蓋掉客戶端自帶的同名標頭，只讓 Kerberos 驗到的帳號傳進來（沒這行＝任何人都能偽造身分）。
  2. 下方「信任的反向代理 IP」要填「實際直接連進本服務的那台反向代理」的 IP。若是「瀏覽器 → 反向代理A → 反向代理B → 本服務」的多層架構，要填最後那台（直接連本服務的 B）的 IP。除非 Nginx 與本服務在同一台，否則不要只留 127.0.0.1。
  3. 本服務只綁 127.0.0.1，或用防火牆只允許反向代理連 8765。否則有人可繞過反向代理、直接對本服務送 X-Remote-User 冒充任何帳號。

提示：你這次開這個頁面，本服務看到的直接來源 IP＝{{ peer_ip or '(未知)' }}。若你正是經反向代理連進來管理的，這通常就是要填到下面清單的值；若你是直接連本服務（未經代理）則不是。

展開 Nginx 設定範例（可複製） {% raw %}
location / {
    # --- SPNEGO / Kerberos（需 libnginx-mod-http-auth-spnego） ---
    auth_gss on;
    auth_gss_realm EXAMPLE.LOCAL;
    auth_gss_keytab /etc/nginx/jtdt.keytab;
    auth_gss_service_name HTTP/doc.example.local;
    auth_gss_allow_basic_fallback off;   # 非網域電腦不擋死，落回 /login

    # --- 關鍵：一律覆寫帳號標頭（防偽造） ---
    proxy_set_header X-Remote-User $remote_user;

    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header X-Forwarded-For $remote_addr;
    proxy_pass http://127.0.0.1:8765;   # 本服務只綁 127.0.0.1
}
{% endraw %}
